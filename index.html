Three ways customers experience Elastic

At Elastic we intentionally tailor capabilities across three complementary experiences (sometimes called “3X UX”). The experiments in this demo mostly illustrate the in-app Kibana and Agent Builder chat surfaces; the same platform primitives should also remain usable from external agents with clear APIs and contracts.

1 In-app Kibana

▣ In-app

Users navigate the product: lists, detail views, charts, and workflows stay anchored in the Kibana UI. Context is discovered where you are in the app.

2 Agent Builder chat

◎ Chat

Users work in a conversational surface inside Elastic. Context, tools, and suggested actions are brought into the thread so they do not have to assemble everything by clicking through every screen first.

3 External agents

↗ External

Customers interact with Elastic from Cursor, Claude, or their own non-Elastic-specific agents. There is no Kibana chrome: integrations rely on explicit, machine-consumable contracts (auth, tools, schemas, outcomes).

Tier 1 explorations (next milestone)

Among everything in this demo, we are prioritizing two areas for near-term product exploration: they deliver clear user value across the experience channels shown on each card, and rest on shared platform primitives (rules, context, episodes).

Onboarding

▣ In-app ↗ External

Rule Onboarding should take users from Streams-aware context through Proposed Alerts, then Create with defaults or Create with agent, all the way to everything required for real alerting: the rule itself, plus sensible defaults for notification policies and workflows so something actually fires and routes when conditions are met. This is the spine for turning analysis into durable, notifyable alerting without exposing every internal building block.

Episode triaging

◎ Chat ↗ External

Episode Triage and Analysis shows how responders move from an active episode to an in-thread investigation: timeline, metrics, evidence, and guided next steps — including visualization grounded in the source index. This is the spine for trustworthy post-fire workflows alongside rules.

Tier 2 explorations

After Tier 1 (onboarding and episode triage) lands, this wave splits into two tracks. Both assume we have already learned how proposals, indices, and guardrails behave in the real product — including the emergence of reusable rule templates from onboarding-style flows.

Rule management

▣ In-app ↗ External

The Rule Doctor tab sketches how agents help operators maintain large rule estates without drowning in noise.

  • Rule coverage — where data patterns are not yet protected, or where new gaps appear after deployments and traffic shifts.
  • Rule tuning — stale thresholds, duplicate or overlapping rules, flapping, and “rule on rule” hygiene so alerts stay actionable.
  • Rule migration — translating, validating, and moving rule definitions across stacks, spaces, or major versions with confidence checks.

Rule authoring in chat

◎ Chat ↗ External

The Rule Authoring tab prototypes chat-led creation in product; Tier 2 work targets the same flows in external agents too (Cursor, Claude, custom) with shared tools and contracts. Natural language to ES|QL, thresholds, and schedules. We stage it in Tier 2 (not Tier 1) because conversational authoring benefits disproportionately once onboarding explorations have defined how proposals, streams context, and review steps behave — and once rule templates exist as concrete starting points (patterns distilled from promoted and well-understood rules) instead of every session inventing structure from scratch.

Tier 3 explorations

After core in-product flows mature, the next horizon is meeting responders in the collaboration surfaces they already live in — without forcing a context switch to Kibana for every step.

Episode triaging (in Slack)

◎ Chat ↗ External

Tier 1’s Episode Triage and Analysis tab focuses on investigation inside Elastic (lists, detail, and an in-thread agent). Tier 3 extends the same intent to Slack: alert and episode context appears in channel or thread, responders can ask follow-ups and kick off actions from there, and deep links keep Kibana one click away when full UI is needed. Product work spans Slack app surfaces, Block Kit / message layout, identity and scopes, and parity with the tools episode triage already uses.

Note for viewers: This page is a static prototype for discussion. Tabs below follow exploration priority: rule onboarding and episode triage in product (Tier 1), then rule management and rule authoring (Tier 2); Tier 3 is episode triage in Slack (not fully mocked here), then RCA-style flows — as if you were in product UI, chat, or Slack; parity for other external agents is a separate product and API exercise.

Rule Onboarding from Streams

Streams continuously samples and inspects your data so Elastic understands what “normal” looks like for each stream and where attention may be needed. That internal analysis feeds our agents; you do not manage or name those building blocks here.

Our opportunity is to layer alert rules on top: actionable rules that trigger notification policies and workflows. Onboarding should carry people through all the content needed to alert — not only the rule definition, but also default notification policies and workflows so responders get notified and playbooks can run without a separate setup project. Using the same data introspection, we propose alerts and let users create them with defaults or create with agent when ready. Either path opens the rule creation form pre-filled with the configuration for that Proposed Alert; with agent, they refine it in the Agent Builder sidebar before saving.

AI Recommendations

Agent Builder has analyzed your rules and identified these optimization opportunities.

✨
Threshold Tuning CPU utilization threshold is too low High container CPU utilization
The current threshold of 80% has generated 23 episodes in the last 7 days, with 18 auto-recovering within 5 minutes. Analysis of baseline CPU patterns shows normal utilization ranges between 65–82% during peak hours. Raising the threshold to 85% would reduce noise by ~60% while still catching sustained spikes.
✨
Duplicate Detection Overlapping rules detected 2 rules affected
CPU threshold > 90% (V1) and High container CPU utilization (V2) monitor the same metric on overlapping infrastructure. The V1 rule uses a static 90% threshold on system.cpu.total.pct, while the V2 rule uses an ES|QL query against k8s.container.cpu_request_utilization. The V2 rule provides better coverage with group-by and episode tracking. Consider decommissioning the V1 rule.
✨
Coverage Gap No memory utilization rules Suggested new rule
You have CPU and restart monitoring, but no rules for memory utilization. The cart-service pod (ep-004) experienced OOMKilled events that could have been predicted with a memory threshold rule. Agent Builder can draft a rule using k8s.container.memory_request_utilization from the same kubelet stats data you already collect.
✨
Threshold Tuning Latency threshold may be too high High service latency (P95)
The P95 latency rule is set at 1000ms, but the recommendation service baseline is only ~350ms. By the time latency crosses 1s, the degradation is severe and users are impacted. A threshold of 700ms would give responders an earlier warning while staying well above normal variance.
✨
Stale Rule Disk usage rule has not fired in 90 days Disk usage > 85% (V1)
The Disk usage > 85% rule is currently disabled and has not generated any alerts in over 90 days. The underlying inventory data source shows current disk utilization across all hosts ranges between 32–58%. This rule may no longer be relevant, or the threshold needs adjustment to match current infrastructure.

Productionizing Agentic Workflows

These experiments are designed as quick wins behind feature flags. Below are the key concerns that must be addressed to make them production-ready.

🔒

Security & Authorization

•
Agent execution identity — Auto-triage runs without a user request. The stored API key's privilege scope must follow least-privilege: only access indices in the rule's FROM clause.
•
Tool-level authz — Each tool call (get_episode_events, get_rule, create_visualization) needs its own authorization check beyond the API key.
•
Artifact ownership — When the agent creates runbooks or dashboards, who owns them? The auto-triage key or the reviewing user? Affects Spaces & RBAC.
•
Prompt injection — Episode data (pod names, log messages) flows into agent prompts. User-controlled strings could be interpreted as instructions.
•
Slack credentials — Bot token and signing secret are in kibana.yml today. Multi-tenant / Serverless needs per-org encryption via ESO.
📈

LLM Evaluation & Quality

•
Evaluation dataset — Build a corpus of episodes with known root causes (CPU ramp, OOMKill, latency cascade) into Synthtrace scenarios for reproducible evaluation.
•
Triage quality scoring — Define "good": correct root cause, followed runbook steps, actionable recommendation. Use Phoenix/Arize tracing to capture and score.
•
Skill reliability — The episode visualization skill has a multi-step workflow. Each step can fail. Need evaluation coverage for CCS indices, multiple FROM sources, derived metrics.
•
Hallucination detection — Factual claims (e.g. "deployment at 08:28") must be traceable to tool call results. Consider a verification step.
•
Regression testing — Prompt/skill/schema changes can silently degrade quality. CI-integrated eval suite on PRs should catch regressions.
📡

Telemetry & Observability

•
Feature adoption — Track auto-triage enablement rate, episodes triaged, and user actions (continue investigation, dismiss, copy summary, share to Slack).
•
Agent performance — Time from episode active → analysis complete, token usage per triage, tool call counts, failure rates.
•
Quality signals — User acknowledged after AI summary (positive) vs. opened full investigation anyway (negative). Thumbs up/down feedback.
•
Cost tracking — Tokens-per-triage and cost-per-episode. Analysis frequency (hourly/daily/weekly) directly impacts LLM spend.
⚡

Reliability & Error Handling

•
Rate limiting & queuing — 100 simultaneous episodes can't spawn 100 parallel agent calls. Use Task Manager with concurrency limits.
•
Timeout & cancellation — Hard ceiling (e.g. 60s) for auto-triage. Graceful degradation to "analysis timed out."
•
Partial results — If the chart skill succeeds but RCA fails, show what you have. Don't discard a good timeline because the summary errored.
•
Idempotency — If the dispatcher re-runs while auto-triage is in progress, don't duplicate. Episode ID + "triage in progress" state prevents this.
•
Stale analysis — Triage from 5 minutes ago may be outdated if the episode evolved. Show staleness indicators and offer re-run.
🗃

Data Model & Storage

•
Analysis storage — Analyses should live in a dedicated index (.alerting-v2-triage-*), not inline on episodes. Enables versioning, search, and independent retention.
•
Schema versioning — Store agent version and skill versions used. Flag outdated analyses when the agent improves and optionally re-run.
•
Artifact linkage — Rules should reference runbook and dashboard saved objects by ID. The agent discovers and reads them as first-class relationships.
•
Retention policy — How long do triage analyses persist? Should they follow episode TTL or have their own lifecycle?
💬

UX & Feedback Loops

•
Thumbs up/down — Every auto-triage summary needs a feedback mechanism. Feed corrections back into the evaluation dataset.
•
Edit & correct — Let users edit wrong root cause summaries. Store corrections as training signal for future eval runs.
•
Confidence indicators — Not all triages are equal. Show real confidence based on evidence strength, not fabricated percentages.
•
Slack message limits — Full summaries may exceed Slack's character limits. Use progressive disclosure: one-line summary with "View full analysis" linking to Kibana.
🌐

Multi-tenancy & Spaces

Space isolation
Auto-triage analyses must be scoped to the Space the rule lives in. Users in Space A must not see triage results from Space B.
Connector configuration
The agent needs an LLM connector. Currently the analysis settings are global (agentic-analysis-global). May need per-space overrides.
Serverless considerations
ESO encryption, API key management, and feature tier gating (is auto-triage Premium?) all need Serverless-specific handling.
